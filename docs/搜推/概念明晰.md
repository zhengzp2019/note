# 概念明晰

## 索引

本文将索引相关概念分为三层：

1. 索引基础对象：`document`、`docid`、`field`、`term`、`posting list`
2. 字段级索引类型：`PRIMARY_KEY64`、`ATTRIBUTE`、`TEXT`、`SUMMARY`
3. 索引组织形式：`KV`、`KKV`、`正排索引`、`倒排索引`

解释了“数据是什么”“字段如何使用”“索引如何组织”三个层面区分开来。

### 索引基础对象

#### Document

`document` 是索引系统中的最小内容单元，即一条可被检索、排序和展示的数据记录。

在不同业务场景中，document 对应的对象有所不同：

1. 电商搜索中通常是一件商品
2. 内容搜索中通常是一篇文章、一条帖子或一个视频
3. 本地生活场景中通常是一个商家或一个 POI

一个 document 通常由“唯一标识 + 多个字段”组成，例如：

```text
docid=1001
title="小米手机充电器"
category="数码"
price=39
brand="小米"
publish_time="2026-03-01"
```

#### docid

`docid` 是 document 在索引系统内部使用的唯一标识。

它与业务主键并不一定完全一致：

1. 业务侧可能命名为 `item_id`、`article_id`、`poi_id`
2. 索引系统通常会将其映射为内部 `docid`，以便于压缩存储、排序和检索

正排索引通常按 `docid` 读取字段，倒排索引中的 posting list 也通常记录
`docid`。

#### Field

`field` 是 document 的字段，是构成 document 的属性。

field 可以从两个维度进行理解：

1. 业务语义维度：字段本身表示什么业务含义，例如 `title`、`price`、
   `category`、`publish_time`
2. 索引语义维度：字段进入索引系统后以何种方式被加工、存储和使用，例如
   `PRIMARY_KEY64`、`ATTRIBUTE`、`TEXT`、`SUMMARY`

前者回答“这个字段是什么”，后者回答“这个字段在索引系统中承担什么职责”。

常见的 field 业务类型包括：

1. 文本字段：`title`、`content`
2. 数值字段：`price`、`score`
3. 枚举或标签字段：`category`、`brand`、`tags`
4. 时间字段：`publish_time`
5. 复杂字段：embedding、多值属性、扩展 JSON

同一个 field 可以同时对应多种索引类型。例如，`title` 可以同时作为 `TEXT`
字段参与全文检索，也可以作为 `SUMMARY` 字段用于结果展示。因此，索引类型并非
field 的天然属性，而是系统赋予 field 的使用方式。

#### Term

`term` 是倒排索引中的最小检索单元。

它通常来自以下来源：

1. 文本分词结果，例如 `title="小米 手机 充电器"` 会切出 `小米`、`手机`、
   `充电器`
2. 枚举字段值，例如 `brand=小米`、`category=耳机`
3. 归一化或改写后的词项，例如同义词、拼写纠错后的词

query 进入搜索系统后，通常也会先被处理为多个 term，再据此查询倒排索引。

#### Posting List

`posting list` 是某个 term 对应的命中文档列表，即“该 term 出现于哪些
document 中”。

最简单的 posting list 仅包含 `docid`，但真实系统中通常还会附带更多信息：

1. `tf(term frequency)`：词频，即该 term 在 document 中出现的次数
2. `position`：位置，即该 term 在 document 中出现的位置
3. `payload/weight`：额外权重或业务附加信息

例如：

```text
term = 手机
posting list = [1, 2]
```

这表示 `docid=1` 和 `docid=2` 这两个 document 均包含 `手机` 这一 term。

### 字段级索引类型

`PRIMARY_KEY64`、`ATTRIBUTE`、`TEXT`、`SUMMARY` 等类型，描述的是 field
在索引系统中的用途，而不是 field 的数据来源。它们并不表示字段来自数据库、
消息队列或接口，而是表示该字段进入索引系统后将被如何处理。

#### Field 与索引类型的关系

| 概念            | 含义                  | 关注点                                    |
| --------------- | --------------------- | ----------------------------------------- |
| `field`         | document 中的业务字段 | 该字段表达什么业务含义                    |
| `PRIMARY_KEY64` | 主键索引类型          | 是否支持按主键快速定位 document           |
| `ATTRIBUTE`     | 属性索引类型          | 是否支持排序、过滤、聚合和按 `docid` 取值 |
| `TEXT`          | 文本索引类型          | 是否支持分词、倒排召回和全文检索          |
| `SUMMARY`       | 摘要存储类型          | 是否支持结果展示时直接返回字段内容        |

可见，field 表示“业务数据本身”，索引类型表示“field 在索引系统中以什么方式被
加工、存储和使用”。

#### PRIMARY_KEY64

`PRIMARY_KEY64` 表示 64 位主键索引，通常用于按主键直接定位 document。

典型用途：

1. 已知业务主键时快速定位 document
2. 增量更新或删除时查找目标 document
3. 主键去重、存在性判断

它更接近“主键查找能力”，而不是全文检索能力。

#### ATTRIBUTE

`ATTRIBUTE` 表示属性索引或属性存储，通常用于保存结构化字段，以支持按
`docid` 读取字段值。

典型用途：

1. 排序取值，例如 `price`、`score`、`hits`
2. 过滤判断，例如 `category`、`status`、`publish_time`
3. 聚合统计，例如按类目、品牌、时间分桶统计

它更接近结构化字段的读取与计算能力。

#### TEXT

`TEXT` 表示文本索引类型，通常会对字段进行分词并建立倒排索引。

典型用途：

1. 标题检索
2. 正文检索
3. 相关性打分、短语匹配、召回

它是全文搜索最核心的字段级索引类型之一。

#### SUMMARY

`SUMMARY` 表示摘要存储类型，通常用于在查询命中后直接返回展示字段。

典型用途：

1. 返回标题、摘要、作者、时间等展示内容
2. 生成搜索结果页中的展示信息
3. 与高亮、snippet 拼接配合使用

它更偏向展示返回，而不是召回或排序。

#### 基于 Schema 的示例说明

以下示例来自
`/Users/zhengzhanpeng.1029/code/java/tianji-ops/tianji-ops-controller/src/main/resources/schema/in0_schema.json`。

该 schema 中定义了如下字段：

```json
"columns": [
  { "name": "title", "type": "TEXT" },
  { "name": "subject", "type": "TEXT" },
  { "name": "id", "type": "UINT32" },
  { "name": "hits", "type": "UINT32" },
  { "name": "createtime", "type": "UINT64" }
]
```

这说明：

1. `title`、`subject` 是业务上的文本字段
2. `id` 是业务主键字段
3. `hits` 是数值字段，可用于热度排序
4. `createtime` 是时间字段，可用于时效性排序或过滤

而在 `indexes` 配置中，这些 field 又被赋予了不同的索引用途：

```json
{ "name": "id", "index_type": "PRIMARY_KEY64" }
{ "name": "id", "index_type": "ATTRIBUTE" }
{ "name": "hits", "index_type": "ATTRIBUTE" }
{ "name": "createtime", "index_type": "ATTRIBUTE" }
{ "name": "title", "index_type": "TEXT" }
{ "name": "subject", "index_type": "TEXT" }
{ "name": "summary", "index_type": "SUMMARY" }
```

可以据此作如下理解：

| field                              | 业务含义       | 索引类型        | 索引用途                                |
| ---------------------------------- | -------------- | --------------- | --------------------------------------- |
| `id`                               | 文档唯一标识   | `PRIMARY_KEY64` | 按主键快速定位 document                 |
| `id`                               | 文档唯一标识   | `ATTRIBUTE`     | 按 `docid` 读取该字段，供返回或处理使用 |
| `hits`                             | 热度值         | `ATTRIBUTE`     | 用于排序、过滤或聚合                    |
| `createtime`                       | 创建时间       | `ATTRIBUTE`     | 用于时效性排序与时间过滤                |
| `title`                            | 标题文本       | `TEXT`          | 分词后参与全文检索                      |
| `subject`                          | 正文或主题文本 | `TEXT`          | 分词后参与全文检索                      |
| `title/subject/id/hits/createtime` | 展示所需字段   | `SUMMARY`       | 查询命中后直接返回展示内容              |

从该例可以看出，同一个 field 可以对应多个索引类型。例如 `id` 同时被配置为
`PRIMARY_KEY64` 和 `ATTRIBUTE`，分别承担“主键定位”与“属性取值”两种职责。
同样，`title` 既可以作为 `TEXT` 字段参与检索，也可以出现在 `SUMMARY` 中用于
展示。

### 索引组织形式

这一层关注的是索引整体如何组织数据，以及查询请求如何访问数据。它与前述
“字段级索引类型”并非同一概念。

#### KV（Key-Value）

KV 的数据模型是 `key -> value`，典型能力是按 `key` 点查。

常见使用场景：

1. 用户画像或用户配置：`uid -> {age, gender, tags...}`
2. 物料基础信息：`docid -> {title, author, category...}`
3. 在线特征或中间结果缓存：`(uid, docid) -> score/features`

典型工程关注点：

1. key 的基数与热点
2. value 的大小与更新频率
3. 是否需要 scan 能力

#### KKV（Key-Key-Value）

KKV 可以理解为“一级主键 + 二级主键”的聚合结构：`pkey -> (skey -> value)`。

常见使用场景：

1. 用户行为序列：`uid(pkey) -> ts(skey) -> event(value)`
2. 同一实体的多版本或多维明细：`item_id(pkey) -> variant_id(skey) -> variant(value)`
3. session 事件流：`session_id(pkey) -> seq(skey) -> event(value)`

典型访问方式：

1. `pkey` 点查取一组数据
2. `pkey + skey` 精确获取单条数据
3. `pkey` 下按 `skey` 范围扫描或取 TopN

典型工程关注点：

1. 控制单个 `pkey` 下的数据量，避免大 key
2. 保证 `skey` 具备明确的排序语义
3. 根据场景设计 TTL、时间回收与冷热分层

#### 正排索引（Forward Index）

正排索引的核心组织方式是：`docid -> 字段值`，即“已知文档，反查该文档有哪些
字段内容”。

在搜推系统中，正排索引通常承担“取数”职责，而不是“找候选”职责。召回、粗排、
精排拿到一批 `docid` 后，需要通过正排索引快速读取这些文档的字段值，用于：

1. 结果展示：标题、摘要、封面、作者、价格、类目等
2. 排序特征：质量分、热度、CTR 预估特征、时效性特征等
3. 过滤判断：状态位、上下架标记、内容安全标记、库存等

它的特点是：

1. 按 `docid` 取字段高效
2. 适合批量读取一批文档的列值
3. 不适合直接回答“哪些 document 包含 term A”这类问题

#### 倒排索引（Inverted Index）

倒排索引的核心组织方式是：`term -> posting list`，即“已知一个词，反查哪些
document 包含这个词”。

在搜索链路中，倒排索引通常承担“召回”和“过滤”职责。query 经过分词、归一化、
同义词改写后，会拿每个 term 去查 posting list，再做交集、并集、打分或截断，
得到候选 document 集合。

常见能力：

1. 全文检索：标题、正文、评论等文本字段检索
2. 属性过滤：品牌=小米、类目=耳机、城市=上海
3. 短语或邻近匹配：依赖 position 信息判断词序和距离

它的特点是：

1. 擅长从海量 document 中快速筛出候选
2. 适合多条件检索与布尔过滤
3. 写入、合并、压缩通常比正排索引更复杂

### 查询流程示例

下面以一个简单示例说明“倒排召回 + 正排取数”的配合方式。

假设有 3 篇 document，只看两个字段：`title` 与 `price`。

```text
docid=1: title="小米 手机 充电器", price=39
docid=2: title="苹果 手机 充电线", price=79
docid=3: title="小米 耳机", price=199
```

#### 正排视角

正排索引可表示为：

```text
1 -> {title: "小米 手机 充电器", price: 39}
2 -> {title: "苹果 手机 充电线", price: 79}
3 -> {title: "小米 耳机", price: 199}
```

当系统已经知道 `docid=2` 时，可以直接读取其 `title` 与 `price`，用于结果展示
或排序特征计算。

#### 倒排视角

若对 `title` 字段分词，可得到：

```text
doc 1 分词: [小米, 手机, 充电器]
doc 2 分词: [苹果, 手机, 充电线]
doc 3 分词: [小米, 耳机]
```

据此建立倒排索引：

```text
小米 -> [1, 3]
苹果 -> [2]
手机 -> [1, 2]
充电器 -> [1]
充电线 -> [2]
耳机 -> [3]
```

#### 一次查询如何执行

查询 `q="手机 充电"`，分词后得到 `[手机, 充电]`。

1. 用倒排索引查每个 term 的候选 document：
   - `手机 -> [1, 2]`
   - `充电 -> []`
2. 根据检索策略对候选集合做交并处理：
   - 若采用 OR 策略，则候选集合为 `[1, 2]`
3. 拿到候选 `docid` 后，再通过正排索引读取展示字段：
   - doc 1: title="小米 手机 充电器", price=39
   - doc 2: title="苹果 手机 充电线", price=79

因此，常见链路可以概括为：

1. 倒排索引负责根据 query 找到候选 `docid`
2. 正排索引负责根据 `docid` 读取字段，用于排序和展示
