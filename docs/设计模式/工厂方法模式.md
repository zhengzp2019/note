# 工厂方法模式

## 一、工厂方法模式(GoF)介绍

### 1. 定义

工厂方法模式的核心思想是：定义一个用于创建对象的接口，但让子类决定实例化哪一个具体类。也就是说，父类并不直接写死具体产品，而是把“创建产品对象”的职责交给子类完成。

### 2. 组成角色

一个较为典型的工厂方法模式，通常包含如下几个角色：

- 抽象产品：定义产品的公共接口
- 具体产品：抽象产品的不同实现
- 抽象创建者：声明工厂方法
- 具体创建者：实现工厂方法，返回具体产品

结合“通知发送”的例子：

- 抽象产品：`MessageSender`
- 具体产品：`SmsMessageSender`、`EmailMessageSender`
- 抽象创建者：`NotificationService`
- 具体创建者：`SmsNotificationService`、`EmailNotificationService`

### 3. 解决的问题

工厂方法模式主要解决的是“当系统存在多种产品实现时，如何避免在父类或调用方中写死具体类”的问题。

如果直接在业务代码里写：

```java
MessageSender sender = new EmailMessageSender();
sender.send("a@example.com", "hello");
```

那么调用方已经和具体实现类耦合在一起了。后续如果希望增加短信、站内信、钉钉机器人等实现，就需要不断修改调用代码。

而使用工厂方法模式之后，调用方只依赖抽象类型，具体创建逻辑由子类决定，==扩展新的产品时，往往只需要增加新的具体创建者和具体产品类。==

### 4. 对比简单工厂

工厂方法模式并不是在一个工厂类里用 `if...else` 或 `switch` 判断类型，而是让不同子类分别承担对象创建职责。因此，它更符合开闭原则。新增产品时，通常只需要新增子类，而不必改动原有工厂类中的条件分支。

不过，这种模式也会带来类数量增加的问题。如果场景非常简单，简单工厂往往更直接；如果系统需要长期扩展，工厂方法模式的可维护性会更好。

## 二、具体示例

**场景说明：** 假设系统需要支持短信通知和邮件通知，二者都属于“通知发送”这一类业务能力，但底层发送器实现不同。

### 1. 定义抽象产品和具体产品

```java
public interface MessageSender {
    void send(String receiver, String content);
}
```

```java
public class SmsMessageSender implements MessageSender {
    @Override
    public void send(String receiver, String content) {
        System.out.println("发送短信给 " + receiver + "，内容为：" + content);
    }
}
```

```java
public class EmailMessageSender implements MessageSender {
    @Override
    public void send(String receiver, String content) {
        System.out.println("发送邮件给 " + receiver + "，内容为：" + content);
    }
}
```

这里的 `MessageSender` 是抽象产品，`SmsMessageSender` 与 `EmailMessageSender` 是具体产品。

### 2. 定义抽象创建者

```java
public abstract class NotificationService {

    protected abstract MessageSender createMessageSender();

    public void notify(String receiver, String content) {
        MessageSender sender = createMessageSender();
        sender.send(receiver, content);
    }
}
```

这个抽象类中最关键的方法是：

```java
protected abstract MessageSender createMessageSender();
```

这就是“工厂方法”。它体现了“工厂”这个词的本意：谁负责创建产品对象，谁就承担了工厂职责。这里父类并不指定到底创建短信发送器还是邮件发送器，而是只声明一个创建入口，把具体创建职责交给子类。

需要注意的是，`notify()` 定义了一段固定流程，因此这个例子同时带有模板方法模式的味道。这并不矛盾，反而是很常见的组合方式：

- `notify()` 体现了模板方法模式中的流程骨架
- `createMessageSender()` 体现了工厂方法模式中的创建延迟

### 3. 定义具体创建者

```java
import org.springframework.stereotype.Service;

@Service
public class SmsNotificationService extends NotificationService {
    @Override
    protected MessageSender createMessageSender() {
        return new SmsMessageSender();
    }
}
```

```java
import org.springframework.stereotype.Service;

@Service
public class EmailNotificationService extends NotificationService {
    @Override
    protected MessageSender createMessageSender() {
        return new EmailMessageSender();
    }
}
```

这两个类就是具体创建者。它们的职责不是完成发送本身，而是决定“本类所对应的通知能力，应该创建哪一种发送器对象”。

### 4. 在控制层中使用

```java
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

@RestController
@RequestMapping("/notify")
public class NotificationController {

    private final SmsNotificationService smsNotificationService;
    private final EmailNotificationService emailNotificationService;

    public NotificationController(SmsNotificationService smsNotificationService,
                                  EmailNotificationService emailNotificationService) {
        this.smsNotificationService = smsNotificationService;
        this.emailNotificationService = emailNotificationService;
    }

    @GetMapping("/sms")
    public String sendSms(@RequestParam String receiver, @RequestParam String content) {
        smsNotificationService.notify(receiver, content);
        return "sms sent";
    }

    @GetMapping("/email")
    public String sendEmail(@RequestParam String receiver, @RequestParam String content) {
        emailNotificationService.notify(receiver, content);
        return "email sent";
    }
}
```

在调用层面，控制器并不关心 `SmsMessageSender` 或 `EmailMessageSender` 这些具体类，而是只关心“调用哪个通知服务”。至于通知服务内部到底创建了什么发送器，由对应的子类自己决定。

### 5. 这个示例中哪里体现了“工厂”

很多人在第一次接触工厂方法模式时，会有一个很自然的疑问：示例里并没有一个名为 `Factory` 的类，为什么还叫工厂方法模式？

原因在于，“工厂”强调的不是类名，而是职责。只要某个方法承担了“创建具体产品对象”的职责，这个方法就可以被称为工厂方法。

在本例中：

```java
protected MessageSender createMessageSender() {
    return new EmailMessageSender();
}
```

这个方法的职责就是创建 `MessageSender` 产品对象，因此它就是工厂方法。`EmailNotificationService` 之所以可以被视为具体创建者，是因为它决定并返回了具体产品 `EmailMessageSender`。

### 6. 工厂模式适用场景

工厂方法模式尤其适合以下场景：

- 系统中存在一组抽象产品及其多种实现
- 希望将对象创建职责从调用方中抽离
- 后续会不断扩展新的产品类型
- 希望父类依赖抽象，而不是依赖具体产品类

如果场景只是“根据一个字符串返回一个对象”，并且后续变化不大，那么简单工厂通常已经足够；如果系统明确存在长期扩展需求，工厂方法模式会更具表达力。

## 三、几个相关模式的对比

### 1. 静态工厂方法

#### 关注点

静态工厂方法的关注点是“如何提供一个更合理的对象创建入口”。它通常用于替代构造器，以解决构造器命名能力不足、无法隐藏实现、无法控制实例数量等问题。

#### 典型写法

```java
public final class User {
    private final String name;

    private User(String name) {
        this.name = name;
    }

    public static User of(String name) {
        return new User(name);
    }
}
```

调用时：

```java
User user = User.of("alice");
```

#### 容易混淆点

静态工厂方法虽然名字里带“工厂”，但它通常只是一个类内部的创建技巧，并不是 GoF 设计模式中的“工厂方法模式”。前者强调“创建入口如何设计”，后者强调“创建职责如何分配”。

### 2. 简单工厂

#### 关注点

简单工厂的关注点是“由一个集中式入口，根据条件返回不同具体实现”。它本质上是把对象创建逻辑集中在一个地方，便于调用方统一获取对象。

#### 典型写法

```java
public interface MessageSender {
    void send(String receiver, String content);
}

public class SmsMessageSender implements MessageSender {
    @Override
    public void send(String receiver, String content) {
        System.out.println("发送短信给 " + receiver + "，内容为：" + content);
    }
}

public class EmailMessageSender implements MessageSender {
    @Override
    public void send(String receiver, String content) {
        System.out.println("发送邮件给 " + receiver + "，内容为：" + content);
    }
}

public final class MessageSenderFactory {
    private MessageSenderFactory() {
    }

    public static MessageSender of(String type) {
        if ("sms".equalsIgnoreCase(type)) {
            return new SmsMessageSender();
        }
        if ("email".equalsIgnoreCase(type)) {
            return new EmailMessageSender();
        }
        throw new IllegalArgumentException("Unsupported sender type: " + type);
    }
}
```

调用时：

```java
MessageSender sender = MessageSenderFactory.of("sms");
sender.send("13800000000", "hello");
```

#### 容易混淆点

很多人口中的“工厂模式”，其实首先想到的是这种写法。但严格来说，简单工厂并不是 GoF 的 23 种设计模式之一，它更像一种常用编码习惯。它和静态工厂方法也很接近，只是简单工厂通常面向“多个实现类的统一创建”。

### 3. 工厂方法模式

#### 关注点

工厂方法模式的关注点是“将具体产品对象的创建职责延迟到子类”。父类只定义创建接口，具体创建哪一个产品，由不同子类决定。

#### 典型写法

```java
public interface MessageSender {
    void send(String receiver, String content);
}

public class SmsMessageSender implements MessageSender {
    @Override
    public void send(String receiver, String content) {
        System.out.println("发送短信给 " + receiver + "，内容为：" + content);
    }
}

public class EmailMessageSender implements MessageSender {
    @Override
    public void send(String receiver, String content) {
        System.out.println("发送邮件给 " + receiver + "，内容为：" + content);
    }
}

public abstract class NotificationService {
    protected abstract MessageSender createMessageSender();
}

public class SmsNotificationService extends NotificationService {
    @Override
    protected MessageSender createMessageSender() {
        return new SmsMessageSender();
    }
}

public class EmailNotificationService extends NotificationService {
    @Override
    protected MessageSender createMessageSender() {
        return new EmailMessageSender();
    }
}
```

#### 容易混淆点

工厂方法模式容易和简单工厂混淆，因为二者都在创建对象。二者真正的区别在于：

- 简单工厂：由一个工厂类集中决定创建哪个对象
- 工厂方法模式：由不同子类分别决定创建哪个对象

换言之，简单工厂是“一个地方集中判断”，工厂方法模式是“把创建职责分散给子类”。

### 4. 策略模式

#### 关注点

策略模式的关注点不是“创建对象”，而是“封装一组可替换的行为或算法，并在运行时选择其中之一”。它更强调行为差异，而不是创建职责。

#### 典型写法

```java
public interface DiscountStrategy {
    BigDecimal discount(BigDecimal amount);
}

public class VipDiscountStrategy implements DiscountStrategy {
    @Override
    public BigDecimal discount(BigDecimal amount) {
        return amount.multiply(new BigDecimal("0.8"));
    }
}

public class NormalDiscountStrategy implements DiscountStrategy {
    @Override
    public BigDecimal discount(BigDecimal amount) {
        return amount;
    }
}

public class PriceCalculator {
    public BigDecimal calculate(BigDecimal amount, DiscountStrategy strategy) {
        return strategy.discount(amount);
    }
}
```

#### 容易混淆点

如果工厂模式中的产品对象本身就是一组行为实现，那么代码外观确实会和策略模式很像。但二者的关注点不同：

- 工厂方法模式关注“谁来创建对象”
- 策略模式关注“运行时使用哪种行为”

### 5. 模板方法模式

#### 关注点

模板方法模式的关注点是“在父类中定义一个固定流程骨架，把某些可变步骤留给子类实现”。它强调的是流程复用，而不是产品创建。

#### 典型写法

```java
public abstract class DataSyncService {

    public final void sync() {
        loadData();
        transformData();
        saveData();
    }

    protected abstract void loadData();

    protected abstract void transformData();

    protected abstract void saveData();
}
```

#### 容易混淆点

模板方法模式和工厂方法模式经常一起出现。因为父类在定义流程骨架时，完全可以把“创建对象”这一步也设计成一个可由子类实现的方法。此时，一个类中既可能有模板方法，也可能有工厂方法。它们不是互斥关系，而是可以组合使用。

## 四、小结

在模式学习中，最重要的不是记住名字，而是看清它们各自解决的问题：

- 静态工厂方法解决“对象创建入口如何设计”的问题
- 简单工厂解决“如何集中管理对象创建”的问题
- 工厂方法模式解决“如何把具体创建职责延迟到子类”的问题
- 策略模式解决“如何在一组行为之间切换”的问题
- 模板方法模式解决“如何复用流程骨架”的问题
