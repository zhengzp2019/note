创建和销毁对象

用静态工厂方法代替构造器

优点：

  1. 静态工厂方法有名字
  2. 不用每次都创建新的对象，可以缓存
  3. 可以返回所声明的返回类型的任何子类型对象
  4. 在方法每次被调用时，所返回对象的类可以随输入参数的不同而改变。

使用场景

1. 需要让创建动作具备明确语义时

构造器只能依靠参数列表区分含义，而静态工厂方法可以通过方法名直接表达意图。尤其是在一个类型存在多种创建方式时，使用 of、from、valueOf、newInstance 等命名，往往比重载构造器更直观。

public final class User {
    private final String username;
    private final String phone;

    private User(String username, String phone) {
        this.username = username;
        this.phone = phone;
    }

    public static User fromUsername(String username) {
        return new User(username, null);
    }

    public static User fromPhone(String phone) {
        return new User(null, phone);
    }
}

调用端仅从方法名就可以看出对象是通过用户名还是手机号创建出来的：

User user1 = User.fromUsername("alice");
User user2 = User.fromPhone("13800000000");

2. 需要复用对象、避免重复创建时

如果某些对象本身是可共享的，或者创建成本较高，那么静态工厂方法可以在内部维护缓存，而不必在每次调用时都返回一个全新的实例。这一点是构造器做不到的。

这里所说的“缓存”主要有两种常见形式：

  1. 单例复用：整个类在运行期间只保留一个实例，对外重复返回同一个对象。
  2. 按条件缓存：并不是全局只有一个对象，而是对相同参数、相同配置或相同状态复用已有实例。

前者强调“实例唯一”，后者强调“相同条件下不重复创建”。二者的共同点在于，实例数量不再由调用方决定，而是由类本身控制。

下面这个例子属于“按条件缓存”。对于同一个数据库地址，调用方始终可以拿到同一个连接池对象：

public final class ConnectionPool {
    private static final Map<String, ConnectionPool> CACHE = new ConcurrentHashMap<>();

    private final String url;

    private ConnectionPool(String url) {
        this.url = url;
    }

    public static ConnectionPool of(String url) {
        return CACHE.computeIfAbsent(url, ConnectionPool::new);
    }
}

ConnectionPool pool1 = ConnectionPool.of("jdbc:mysql://localhost:3306/demo");
ConnectionPool pool2 = ConnectionPool.of("jdbc:mysql://localhost:3306/demo");

而下面这个例子则属于“单例复用”：

public final class ConfigCenter {
    private static final ConfigCenter INSTANCE = new ConfigCenter();

    private ConfigCenter() {
    }

    public static ConfigCenter getInstance() {
        return INSTANCE;
    }
}

ConfigCenter c1 = ConfigCenter.getInstance();
ConfigCenter c2 = ConfigCenter.getInstance();

3. 只想暴露抽象类型，不想暴露具体实现时

在很多场景下，调用方只需要依赖接口，不应该感知底层实现类。静态工厂方法可以把实现细节藏在方法内部，只把抽象类型暴露出去，这样后续替换实现也更容易。

public interface Storage {
    void put(String key, String value);
}

public final class Storages {
    private Storages() {
    }

    public static Storage memory() {
        return new InMemoryStorage();
    }

    private static final class InMemoryStorage implements Storage {
        @Override
        public void put(String key, String value) {
            System.out.println("save to memory: " + key + "=" + value);
        }
    }
}

调用方只面向 Storage 编程，而不需要知道实现类名称：

Storage storage = Storages.memory();
storage.put("name", "effective-java");

4. 需要根据参数动态选择不同实现时

有些类型在概念上是同一类能力，但实现方式会随输入参数、运行环境或配置项变化。此时静态工厂方法可以把“选择哪一个实现类”的逻辑集中起来，避免调用方自己编写分支判断。

public interface MessageSender {
    void send(String message);
}

public final class MessageSenders {
    private MessageSenders() {
    }

    public static MessageSender of(String type) {
        if ("sms".equalsIgnoreCase(type)) {
            return new SmsSender();
        }
        if ("email".equalsIgnoreCase(type)) {
            return new EmailSender();
        }
        throw new IllegalArgumentException("Unsupported sender type: " + type);
    }

    private static final class SmsSender implements MessageSender {
        @Override
        public void send(String message) {
            System.out.println("send sms: " + message);
        }
    }

    private static final class EmailSender implements MessageSender {
        @Override
        public void send(String message) {
            System.out.println("send email: " + message);
        }
    }
}

调用方只关心业务参数，而不必自己决定实例化哪个具体类：

MessageSender sender = MessageSenders.of("sms");
sender.send("hello");
